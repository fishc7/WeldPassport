# План документации модуля ОГС (отдел главного сварщика)

Назначение: определить, какие документы по роли «Главный сварщик (ОГС)» нужно
написать, в каком порядке и что в каждый входит. Сами документы создаются по этому
плану отдельными шагами.

## 1. Что уже есть (опора, не переписывать)

- Модель данных «Работники и сварщики» v0.1 — **согласована**:
  `../../10_Проектирование_WeldPassport/03_Работники_и_сварщики/01_Модель_данных_Работники_и_сварщики_v0.1.md`.
  Описывает статус сварщика, клейма, документы, аттестации, внутренние допуски,
  допуски к объекту, проверку назначения.
- План реализации «Работники и сварщики» v0.1 — там же, файл `02_План_реализации…`.
- Код: `09_Разработка/src/models/welders.py`, `workers.py` (таблицы СВАРЩИКИ,
  ДОКУМЕНТЫ_СВАРЩИКА, АТТЕСТАЦИИ_СВАРЩИКОВ, ВНУТРЕННИЕ_ДОПУСКИ_СВАРЩИКОВ,
  ДОПУСКИ_К_ОБЪЕКТУ) и модуль `09_Разработка/backend/app/workforce/`.
- Готовые данные НАКС: SQLite `M_Kran_Kingesepp.db`, таблица `naks_data` (423 строки,
  66 сварщиков, 318 удостоверений) — источник первичного импорта аттестаций.
  Сопоставление полей — в ТЗ ОГС, раздел 5.

## 2. Чего нет (заполняем)

Папка роли `05_Роли_и_права/05_Главный_сварщик_ОГС/` — пустой каркас: `ТЗ`,
`Процессы`, `Права_доступа`, `Прототипы_экранов`, `Backend`, `Frontend`. Файлов нет.

## 3. Что закреплено за ОГС (зона ответственности)

| Объект | Сущность модели / таблица |
|---|---|
| Статус сварщика (назначение/завершение вручную) | Период статуса сварщика / СВАРЩИКИ |
| Клейма — личные и объектовые | Клеймо |
| Документы сварщика (НАКС, диплом, обучение, удостоверение) | ДОКУМЕНТЫ_СВАРЩИКА |
| Аттестации и их области (способ, материал, диаметры, толщины, положение) | АТТЕСТАЦИИ_СВАРЩИКОВ |
| Внутренние допуски | ВНУТРЕННИЕ_ДОПУСКИ_СВАРЩИКОВ |
| Допуски заказчика к объекту | ДОПУСКИ_К_ОБЪЕКТУ |
| Проверка возможности назначения (блокировки/предупреждения) | Раздел 5 модели данных |
| Приём сканов документов и парсинг НАКС с naks.ru | ДОКУМЕНТЫ_СВАРЩИКА + объектное хранилище + интеграция с реестром |

ОГС **не делает**: приём/увольнение и кадровую карточку (это ОК), назначение
сварщиков на стыки и фактическую сварку (Мастер/ПТО), контроль качества (ОТК/НК).

## 4. Документы к созданию (по порядку)

### Шаг 1 — ТЗ модуля ОГС v0.1  → `ТЗ/WeldPassport_ОГС_ТЗ_v0.1.md`

Якорный документ, по образцу ТЗ ОК. Состав:

1. Роль ОГС в системе (что делает / чего не делает).
2. Процессы ОГС (кратко, детали — в шаге 2).
3. Карточки и поля **с дублированием ключевых полей** (по решению — повторить таблицы
   полей из модели данных для удобства чтения; пометить, что канон — модель данных).
4. Проверка возможности назначения (блокирующие и предупреждающие условия).
5. Открытые вопросы.

### Шаг 2 — Процессы ОГС  → `Процессы/WeldPassport_ОГС_Процессы_v0.1.md`

Пошагово по каждому процессу с ответственными и переходами статусов:

- назначение и завершение статуса сварщика;
- закрепление личного и объектового клейма;
- регистрация документа сварщика (НАКС, диплом, обучение);
- внесение аттестации и её областей;
- оформление внутреннего допуска;
- оформление допуска заказчика к объекту;
- проверка возможности назначения на дату и объект;
- парсинг НАКС и импорт `naks_data` — см. готовый документ
  [Парсинг НАКС](Процессы/WeldPassport_ОГС_Парсинг_НАКС_v0.1.md).

### Шаг 3 — Права доступа ОГС  → `Права_доступа/WeldPassport_ОГС_Права_v0.1.md`

RBAC со scope (организация/проект/объект): что ОГС создаёт, меняет, видит; разграничение
с ОК (ОГС редактирует профессиональные сведения, ОК — кадровые); чтение чужих данных.

### Шаг 4 — Прототипы экранов ОГС  → `Прототипы_экранов/`

Карточка сварщика (вкладки: профиль, клейма, документы, аттестации, допуски), списки
с фильтрами по статусу/сроку, формы добавления, индикатор «допущен/не допущен».

### Шаг 5 — Заметки Backend/Frontend  → `Backend/`, `Frontend/`

Привязка к существующему модулю `workforce` (api, services, repository, schemas):
какие команды/эндпоинты нужны под процессы шага 2.

## 5. Принятые решения

- ТЗ **дублирует ключевые поля** из модели данных для удобства; канон остаётся за
  `10_Проектирование/03_Работники_и_сварщики/01_Модель_данных…` — при расхождении
  правит модель, затем ТЗ.
- Каждый новый документ привязывается в `05_Роли_и_права/README.md` и при необходимости
  в `00_НАВИГАЦИЯ.md`.
