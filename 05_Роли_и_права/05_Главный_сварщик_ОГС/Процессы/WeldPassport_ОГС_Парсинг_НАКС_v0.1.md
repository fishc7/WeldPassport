# Парсинг НАКС — пайплайн и интеграция в WeldPassport (v0.1)

Назначение: описать существующий парсер реестра НАКС (из проекта `M_Kran`, код изучен)
и план его встраивания в WeldPassport как источника аттестаций сварщиков.

Источник реестра: <https://naks.ru/registry/personal/>.
Скрипты получены и лежат в `_Входящее/`: `naks_search.py`,
`join_naks_files_excel_merged.py`, `load_naks_to_db.py`.

## 1. Существующий пайплайн (код изучен)

```
naks_search.py (Selenium)
  → naks_главное.xlsx + naks_подробнее.xlsx
  → join_naks_files_excel_merged.py
  → naks_merged.xlsx
  → load_naks_to_db.py
  → naks_data (SQLite)
```

### 1.1 `naks_search.py` — скрейпер (Selenium + BeautifulSoup)

- Chrome, **видимый** (`--start-maximized`, не headless), открывает страницу реестра.
- `time.sleep(30)` — фиксированная пауза на **ручное решение CAPTCHA** (без проверки условия).
- Шаг 1: парсит основную таблицу (`#app_registry_personal … table tbody tr`), берёт
  первые 13 колонок → `naks_главное.xlsx` (ФИО, шифр клейма, место работы, должность,
  номер удостоверения, доп.атт., АЦ, АП, дата аттестации, окончание срока, срок продления,
  вид деятельности, область аттестации).
- Шаг 2: по каждой ссылке «подробнее» открывает модальное окно `#oblast_att`, парсит две
  таблицы `table-detailPersonal`:
  - таблица 1 — пары «ключ: значение» (реквизиты);
  - таблица 2 — строки с 2/3/4 колонками, значения группируются по ключу и склеиваются
    через `" | "` (области аттестации: способ, материал, диаметры, толщины, положение…).
  - результат + заголовок модалки `_УДОСТОВИРЕНИЕ_НАКС_` → `naks_подробнее.xlsx`.
- По завершении сам запускает `join_…` через `subprocess`.

### 1.2 `join_naks_files_excel_merged.py` — объединение

- `pandas.merge(how="left", left_index=True, right_index=True)` — склейка главного и
  подробного **по индексу строк** → `naks_merged.xlsx`.

### 1.3 `load_naks_to_db.py` — загрузка в SQLite

- Читает `naks_merged.xlsx`, чистит имена колонок.
- **Инкрементальность**: ключ дедупликации — `_УДОСТОВИРЕНИЕ_НАКС_` (заголовок модалки);
  строки с уже существующим ключом пропускаются.
- **Динамическая схема**: если таблицы нет — создаёт; если есть — `ALTER TABLE ADD COLUMN`
  под новые колонки (все `TEXT`). Запись через `to_sql(append)` с fallback на `executemany`.
- Зависит от `scripts.core.database.DatabaseConnection` (специфично для M_Kran).

## 2. Ограничения (важно для архитектуры)

- **CAPTCHA + видимый браузер + фиксированный `sleep(30)`** → нельзя запускать
  автоматически на сервере. Это операторский batch-инструмент, не онлайн-API.
- **Хрупкость**: жёсткая привязка к XPath/верстке naks.ru — при изменении сайта ломается.
- **Динамическая «плоская» схема** (`naks_data`, все поля `TEXT`) — противоположность
  структурной модели WeldPassport. Целевая БД — **только PostgreSQL**.
- **Главный изъян потока — три прохода через Excel и объединение по индексу строк:**
  - `merge(left_index, right_index)` сопоставляет «главное» и «подробнее» по **порядку
    строк**. Любой пропуск, повторный поиск ссылок (`StaleElement`) или сбой на одной
    модалке сдвигает соответствие — данные молча перепутываются;
  - три файла Excel = лишний ввод-вывод, потеря типов, проблемы кодировок, лишние точки
    отказа;
  - нет транзакции: частичный сбой оставляет неконсистентные данные.

## 3. Целевая архитектура (переработка)

Парсер **требует переработки**, а не переноса. Excel-промежуточники и объединение по
индексу убираем. Принцип: **парсить сразу в структурированную запись и писать прямо в БД,
одним проходом**.

Было (неправильно):

```
главное → Excel1 ┐
                 ├─ merge по индексу → Excel3 → плоская TEXT-таблица
подробнее → Excel2 ┘
```

Станет (целевое):

```
для каждой записи реестра:
  читаем основные поля строки
  открываем «подробнее», читаем модалку
  собираем ОДНУ полную запись (объект) — главное и детали уже связаны, без join
  нормализуем (даты ДД.ММ.ГГГГ → date, многозначные поля разбиваем)
  upsert в структурные таблицы PostgreSQL в транзакции
```

Раскладка одной записи:

- сварщик — сопоставление по **ФИО** с `РАБОТНИКИ`/`СВАРЩИКИ` (контроль дублей, модель §6);
- удостоверение → `ДОКУМЕНТЫ_СВАРЩИКА` (Вид = НАКС, номер, даты, ссылка на скан);
- области аттестации → `АТТЕСТАЦИИ_СВАРЩИКОВ` (способ, материал, диапазоны, положение,
  тип соединения, группа тех. устройств);
- происхождение записи — фиксировать (источник = реестр НАКС, дата прогона) по модели §7.

Преимущества: нет рассинхрона главное↔детали (они связаны в момент парсинга), нет
Excel-файлов, есть типы и транзакционная целостность, дедуп по **номеру удостоверения**.

Что сохраняем из старого кода: логику Selenium-навигации и разбора модального окна
(`#oblast_att`, две таблицы) — она рабочая; меняем только то, куда и как пишем результат.

Стек: Selenium + SQLAlchemy, модуль `backend/app/workforce`. Скрейпер — операторский
скрипт в `09_Разработка/scripts/naks/` (из-за CAPTCHA).
