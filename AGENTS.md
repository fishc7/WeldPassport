# AGENTS.md — инструкция для AI-агентов проекта WeldPassport

Этот файл — руководство для AI-агентов (Cursor и др.), работающих в репозитории.
Он описывает, что это за проект, как устроен репозиторий, как запускать код и какие
правила нельзя нарушать. Личный профиль владельца — в `Context/AboutMe.md`.

---

## 1. Что это за проект

**WeldPassport** — веб-приложение для отдела главного сварщика (ОГС) и управления
сварочным производством трубопроводов.

Главный принцип системы:

> WeldPassport фиксирует не только исполнительную документацию, но и **фактическое
> выполнение** сварочного производства.

Это не просто журнал сварки, а MES/ERP-система для сварочного производства.
Центральная сущность модели — **стык** (joint); проект, объект и изометрия задают
его контекст, а производство и контроль формируют историю исполнения.

## 3. Технологический стек

**Текущее состояние кода:** Python + SQLAlchemy 2 + psycopg (raw) + python-dotenv,
PostgreSQL. Это пока модели и служебные скрипты, без HTTP-слоя.

**Целевой стек MVP** (см. `docs/ARCHITECTURE.md`):

- Backend: Python 3.12, FastAPI, SQLAlchemy 2, Alembic, Pydantic, PostgreSQL 16,
  Celery/Dramatiq + Redis для фоновых задач.
- Frontend: React + TypeScript, Vite, React Query, React Hook Form, Ant Design/MUI.
- Инфраструктура: Docker Compose, Timeweb Cloud, Nginx + TLS, Selectel Object
  Storage (S3-совместимый) для файлов.
- Excel import/export — через openpyxl или pandas.

**Жёсткое правило: основная БД — только PostgreSQL. Не предлагать и не использовать
SQLite в качестве основной БД.**

Архитектура backend — модульный монолит. Предметные модули (`identity`, `projects`,
`engineering`, `workforce`, `admissions`, `production`, `quality`, `documents`,
`reporting`, `audit`) не лезут напрямую в таблицы друг друга — только через сервисы.
Слои: API → Application services → Domain rules → Repository → SQLAlchemy → PostgreSQL.

## 4. Запуск и окружение

Все команды выполняются из каталога `09_Разработка/` (Windows, PowerShell).

```powershell
# 1. виртуальное окружение и зависимости
python -m venv .venv
.\.venv\Scripts\Activate.ps1
pip install -r requirements.txt

# 2. конфигурация БД
Copy-Item .env.example .env   # затем заполнить пароль и параметры

# 3. создание таблиц (idempotent: существующие не трогает)
python scripts/create_tables.py
```

Переменные окружения (`.env`): `POSTGRES_HOST`, `POSTGRES_PORT`, `POSTGRES_DB`,
`POSTGRES_USER`, `POSTGRES_PASSWORD`, `POSTGRES_SCHEMA` (по умолчанию `test`).

Особенности доступа к БД:

- `config.py` читает `.env` из корня `09_Разработка/` и формирует `settings`.
- И `db.py` (psycopg), и `database.py` (SQLAlchemy) выставляют
  `SET search_path TO "<POSTGRES_SCHEMA>", public` на каждом подключении.
- Скрипты в `scripts/` добавляют `src/` в `sys.path` и импортируют из `src`
  (`from config import ...`, `from models import ...`). Запускай их из
  `09_Разработка/` как `python scripts/<имя>.py`.

`.env` закоммичен в `.gitignore` — никогда не коммить секреты.

## 5. Доменные правила (обязательны при моделировании и коде)

Жизненный цикл системы (модули 1 и 2 идут параллельно, затем сходятся):

1. Прием и допуск сварщика
2. Подготовка документации
3. Подготовка производства
4. Передача в работу
5. Фактическое выполнение
6. Контроль
7. Закрытие

**Всегда разделяй роли вокруг стыка** (это разные сущности, даже если в записи их
выполняет один человек):

- официально допущенный сварщик;
- назначенный сварщик;
- фактический сварщик;
- сварщик в исполнительной документации;
- помощник сварщика / звеньевой (подготовка);
- мастер / прораб, подтвердивший факт;
- сотрудник ПТО, внёсший данные.

Ключевые инварианты модели данных:

- Разделять `worker` (работник), `welder_profile` (профиль сварщика) и
  `user_account` (учётка). Не каждый работник имеет учётку и наоборот.
- **Допуск — это результат правил, а не флаг.** Результат проверки сохраняется
  как неизменяемый снимок (`admission_check`), чтобы потом доказать решение.
- **История вместо перезаписи.** Факты, контроль, ремонты и допуски не удаляются
  и не перезаписываются без следа; исправление = новая версия/корректирующая
  запись. Для справочников допустимо мягкое удаление.
- Числа — `numeric` (диаметр, толщина), диапазоны — отдельные `min`/`max`, даты —
  `date`/`timestamptz`, статусы/виды — справочники или enum. Не хранить числа строками.
- Переходы статуса стыка оформляются командами с проверкой условий, а не правкой
  поля. Бизнес-действия в API — команды (`/joints/{id}/record-welding` и т.п.),
  а не универсальный update.
- Файлы — в объектном хранилище; в PostgreSQL только метаданные.
- Авторизация — RBAC со scope (организация/проект/объект); проверка прав строго на
  backend, скрытие кнопки во frontend защитой не считается.

## 6. Уточняющие вопросы (обязательно)

Перед тем как писать код, менять схему БД или архитектуру, **задавай уточняющие
вопросы**, если в задаче есть неоднозначность. Лучше один раз уточнить, чем
переделывать.

Когда обязательно спрашивать:

- требования размыты, не хватает данных или возможно несколько трактовок;
- есть несколько разумных вариантов реализации с разными последствиями;
- задача затрагивает модель данных, миграции, статусы стыка, допуски или роли;
- изменение влияет на несколько модулей или ломает обратную совместимость;
- предстоит удаление данных, перезапись истории или другое необратимое действие.

Как спрашивать:

- задавай вопросы **до** начала работы, а не после;
- формулируй конкретно, по возможности предлагай 2–3 варианта с плюсами/минусами
  и помечай рекомендуемый;
- группируй несколько вопросов вместе, не растягивай на много сообщений;
- по мелочам (именование, формат, выбор среди равнозначных вариантов) не спрашивай —
  прими разумное решение и кратко отметь его.

Открытые вопросы, требующие согласования, перечислены в разделе 14
`docs/ARCHITECTURE.md` — учитывай их и поднимай при первой релевантной задаче.

## 9. Приём наработок (входящее)

Владелец не раскладывает новые материалы по папкам вручную — это делает агент.

Порядок:

1. Любой новый материал на разбор кладётся в `_Входящее/` (корень репозитория,
   игнорируется git — можно бросать черновики и выгрузки с ПДн) либо передаётся
   прямо в сообщении.
2. Владелец одной фразой поясняет, что это и к чему относится.
3. Агент определяет место по структуре (§2), создаёт/оформляет файл в нужной папке,
   проставляет ссылки, обновляет навигацию (`00_НАВИГАЦИЯ.md` и при необходимости
   `10_Проектирование/00_Навигация_проектирования.md`) и
   при необходимости — `docs/КАК_РАБОТАТЬ.md`.
4. Агент сообщает о дублях и конфликтах с существующими материалами, не создавая
   копий (каждый документ — в одном основном месте).
5. Перенос разобранного из `_Входящее/` и удаление готовятся командой PowerShell —
   их выполняет владелец (среда не разрешает агенту удаление/перенос).

Сырые материалы для базы знаний Obsidian (статьи, заметки на длительный разбор) —
в `D:\Knowledge_Base\raw\`, а не в `_Входящее/`.
