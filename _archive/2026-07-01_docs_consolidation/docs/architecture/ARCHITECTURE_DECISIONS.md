# Назначение

Документ фиксирует ключевые архитектурные решения проекта WeldPassport.

Каждое решение должно содержать:

* проблему;
* рассмотренные варианты;
* принятое решение;
* последствия.

Архитектурные решения принимаются редко, но оказывают влияние на весь проект.

---

# ADR-0001

## Главная бизнес-сущность

### Проблема

Необходимо определить центральную сущность системы.

### Рассматривались варианты

* Пользователь
* Работник
* Сварщик
* Документ
* Сварной стык

### Решение

Главной сущностью системы является **сварной стык (Weld Joint)**.

Все остальные сущности существуют для создания, сопровождения, контроля или анализа сварного стыка.

### Последствия

* Все бизнес-процессы строятся вокруг жизненного цикла стыка.
* Производственные данные становятся связаны единой моделью.
* Упрощается трассировка и аналитика.

---

# ADR-0002

## Архитектурный стиль

### Решение

Использовать модульную архитектуру (Modular Monolith) как основу проекта.

### Причины

* Простота разработки.
* Высокая производительность.
* Отсутствие сложности микросервисов.
* Возможность дальнейшего выделения сервисов.

### Последствия

Каждый домен реализуется как независимый модуль внутри Backend.

---

# ADR-0003

## Backend

### Решение

Backend разрабатывается на:

* FastAPI
* SQLAlchemy 2.x
* PostgreSQL
* Alembic

### Причины

* Современный стек.
* Асинхронная работа.
* Хорошая масштабируемость.
* Богатая экосистема.

---

# ADR-0004

## Frontend

### Решение

Frontend реализуется отдельно от Backend.

Используются:

* React
* TypeScript
* Vite

### Причины

* Независимое развитие интерфейса.
* Современный стек.
* Быстрая сборка.
* Простая интеграция с REST API.

---

# ADR-0005

## Организация бизнес-логики

### Решение

Вся бизнес-логика располагается исключительно в Backend.

Frontend отвечает только за:

* отображение данных;
* взаимодействие с пользователем;
* вызовы API.

### Последствия

* Один источник бизнес-логики.
* Предсказуемое поведение системы.
* Упрощение сопровождения.

---

# ADR-0006

## База данных

### Решение

PostgreSQL является единственным источником хранения данных.

Все изменения структуры выполняются через Alembic.

### Последствия

* Полная воспроизводимость схемы.
* Контроль версий БД.
* Упрощение развёртывания.

# ADR-0011 — Backend как основная архитектурная база

## Статус

Принято.

## Решение

Backend в каталоге `09_Разработка/backend` принимается как основная архитектурная база WeldPassport.

## Обоснование

Backend уже имеет правильное разделение слоёв:

- `api`
- `schemas`
- `services`
- `repository`
- `models`
- `shared`

Стек проекта:

- FastAPI
- SQLAlchemy 2
- PostgreSQL
- Alembic
- Pydantic Settings

## Последствия

Backend не переписывается с нуля.

В Sprint 0 необходимо выполнить только стабилизацию:

- исправить кодировку UTF-8;
- создать первую Alembic-миграцию;
- добавить базовые тесты;
- определить судьбу `09_Разработка/src`.

# Правила внесения новых решений

Новое архитектурное решение добавляется только если оно:

* влияет на несколько модулей;
* изменяет архитектуру;
* меняет подход к разработке;
* влияет на долгосрочное развитие проекта.

Каждое решение должно иметь собственный идентификатор (ADR-XXXX) и не изменяться задним числом. Если решение пересматривается, создаётся новая запись с ссылкой на предыдущую.
