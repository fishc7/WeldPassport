# Назначение

Документ описывает предметную модель WeldPassport.

Любая новая функциональность должна вписываться в эту модель.

---

# Главная бизнес-сущность

## Weld Joint (Сварной стык)

Вся система строится вокруг сварного стыка.

Именно стык является объектом:

* производства;
* контроля;
* исполнительной документации;
* нормирования;
* аналитики;
* аттестации.

Жизненный цикл стыка определяет жизненный цикл всей системы.

---

# Жизненный цикл сварного стыка

1. Создание объекта строительства.
2. Создание технологической линии.
3. Создание узла.
4. Создание сварного стыка.
5. Назначение исполнителей.
6. Выполнение сварки.
7. Контроль качества.
8. Формирование исполнительной документации.
9. Закрытие стыка.
10. Архив.

# Главные агрегаты

## Aggregate Root

### Worker

Объединяет:

* личные данные;
* должность;
* подразделение;
* историю.

---

### Welder

Объединяет:

* работника;
* клейма;
* допуски;
* аттестации.

---

### Weld Joint

Главный Aggregate Root системы.

Содержит:

* объект;
* линию;
* узел;
* материалы;
* сварщиков;
* операции;
* результаты контроля;
* исполнительную документацию;
* статус.

Все остальные домены так или иначе связаны со сварным стыком.

---

# Статусы сварного стыка

1. Создан.
2. Подготовлен.
3. Собран.
4. Сварен.
5. Проверен.
6. Исправлен.
7. Принят.
8. Закрыт.
9. Архивирован.

---

# Бизнес-правила

* Работник может не быть сварщиком.
* Каждый сварщик обязан быть работником.
* Клеймо принадлежит одному сварщику.
* Допуск имеет срок действия.
* Сварной стык может иметь несколько сварочных операций.
* Контроль проводится только после выполнения сварки.
* Исполнительная документация формируется только после завершения контроля.
* Закрытый стык запрещено изменять без специальной процедуры.

---

# Архитектурные принципы

1. Один источник истины для каждой сущности.
2. Каждая сущность имеет владельца.
3. Все действия пользователей протоколируются.
4. Бизнес-логика располагается только в Backend.
5. Frontend не хранит состояние предметной области.
6. Все изменения проходят через API.
7. Все ключевые сущности имеют историю изменений (Audit Trail).
