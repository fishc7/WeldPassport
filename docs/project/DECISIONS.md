# WeldPassport — журнал проектных решений

## Назначение

Этот файл фиксирует важные проектные и архитектурные решения.

Формат записи — ADR: Architecture Decision Record.

ADR нужен, чтобы через время было понятно:

- какое решение принято;
- почему оно принято;
- какие варианты были отклонены;
- какие последствия у решения;
- где находится каноническое описание.

Канон архитектуры — `docs/ARCHITECTURE.md`; контекст домена —
`docs/00_PROJECT_CONTEXT.md`. Краткая сводка — `docs/project/PROJECT_SUMMARY.md`;
план работ — `docs/project/ROADMAP.md`.

## 2026-07-01 — Завершение архитектурной ревизии

**Дата:** 2026-07-01

**Контекст:** завершена независимая архитектурная ревизия проекта; выводы
подтверждены тремя согласованными анализами (ChatGPT, Cursor Agent, Claude).

**Принятые решения:**

- Backend принят как основная архитектура.
- Документация больше не должна опережать код.
- Начинается реализация вертикальных бизнес-сценариев.
- Первый вертикальный сценарий — полный жизненный цикл сварного стыка.
- ETL откладывается.
- Import Pipeline остаётся внутри Backend.
- Cursor используется для реализации.
- Архитектурные решения принимаются только после ревизии.

**Связанные материалы:** `docs/ARCHITECTURE.md` (раздел «Архитектурная веха —
2026-07-01»).

## 2026-07-01 — Изменение стратегии разработки

**Дата:** 2026-07-01

**Контекст:** после независимой архитектурной экспертизы (ChatGPT, Cursor Agent,
Claude) пересмотрен порядок вертикальной реализации. Ранее первым сценарием
предполагался полный жизненный цикл сварного стыка (см. запись «Завершение
архитектурной ревизии» выше).

**Принятые решения:**

- Мы изменяем порядок разработки.
- Первым вертикальным сценарием становится полный жизненный цикл работника.
- Вторым вертикальным сценарием станет полный жизненный цикл сварного стыка.
- Контур допуска является фундаментом производственной системы WeldPassport.
- Это решение не отменяет центральную роль сварного стыка в производственном
  контуре.

**Причина:** контур допуска является фундаментом всей производственной системы
WeldPassport.

**Связанные материалы:** `docs/ARCHITECTURE.md` (подраздел «Приоритет первого
вертикального сценария» в разделе «Архитектурная веха — 2026-07-01»).

## ADR-001. Модель организаций и проектов

Дата: 2026-07-02

Статус: принято

### Контекст

В WeldPassport необходимо учитывать реальные строительные и сварочные проекты, где
в одном проекте могут участвовать разные организации: заказчик, генподрядчик,
сварочный подрядчик, лаборатория НК, технадзор, проектировщик и другие участники.

Модель «1 фирма = 1 проект» не подходит, потому что она ограничивает систему и не
отражает реальную структуру проекта.

### Решение

В системе используется модель связи «многие ко многим»:

```text
companies ↔ projects
```

Связь реализуется через таблицу:

```text
project_companies
```

Минимальные поля:

```text
project_id
company_id
role
```

Одна организация может участвовать во многих проектах.

Один проект может включать несколько организаций.

Роль организации в проекте фиксируется в поле `role`.

Примеры ролей:

```text
customer              — заказчик
general_contractor    — генподрядчик
welding_contractor    — сварочный подрядчик
ndt_lab               — лаборатория НК
inspection            — технадзор / инспекция
designer              — проектировщик
```

### Последствия

Модель позволяет корректно вести проекты с несколькими участниками.

В дальнейшем все модули WeldPassport должны использовать эту модель при работе с
проектами, организациями, документами, сварщиками, стыками, НК и исполнительной
документацией.

AI-агенты и разработчики не должны создавать реализацию по принципу «1 фирма = 1
проект».

Нельзя ограничивать проект одним полем:

```text
projects.company_id
```

Если нужно указать участников проекта, используется связующая таблица
`project_companies`.
