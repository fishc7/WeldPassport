# Как работать в репозитории WeldPassport

Краткая памятка по структуре проекта и порядку работы. Канон по сути системы —
`AGENTS.md`, `docs/00_PROJECT_CONTEXT.md`, `docs/ARCHITECTURE.md`.

Единая точка входа по всем материалам — корневой файл `00_НАВИГАЦИЯ.md`. Любой новый
документ нужно добавлять в него (и, если он по проектированию, — в
`10_Проектирование_WeldPassport/00_Навигация_проектирования.md`).

## Где что лежит

| Папка | Назначение | В git? |
|---|---|---|
| `docs/` | Канон: контекст, архитектура, модель данных, схема БД | да |
| `09_Разработка/` | Весь код: `src/` (модели), `scripts/` (рабочие скрипты), `backend/` (FastAPI MVP) | да |
| `00`–`08_*` | Документные материалы: паспорт, процессы, реестр БД, роли, ТЗ, прототипы, справочники | да |
| `10_Проектирование_WeldPassport/` | Рабочие черновики проектирования по темам (ссылаются на `docs/`, без копий) | да |
| `Context/` | Профиль владельца и предпочтения (агенты читают первым) | да |
| `tools/` | Разовые скрипты импорта 1С + логи (кадровые данные) | нет, игнорируется |
| `Работники/` | Табели с персональными данными | нет, игнорируется |

Вне репозитория:

- `D:\Knowledge_Base` — база знаний Obsidian (вынесена из репо).
- `D:\Archive_данные_projects` — спасённые данные из старых проектов `projects_my-app`.

## Куда класть новое

- Документ или решение по архитектуре/БД → `docs/` (единственный источник правды).
- Код приложения → `09_Разработка/`: модели в `src/models/`, рабочие скрипты в
  `scripts/`, API — в `backend/`.
- Черновик/заметка по теме проектирования → нужный раздел `10_Проектирование/`,
  и ставить ссылку, а не копию (правило из `10_Проектирование/00_Навигация_проектирования.md`).
- Данные с ПДн (табели, выгрузки 1С) → `Работники/` или `tools/` — они не попадают
  в git автоматически.

Перед любой задачей агент читает `AGENTS.md`, `docs/00_PROJECT_CONTEXT.md`,
`docs/ARCHITECTURE.md` и `Context/`. Там зафиксированы доменные правила: разделение
ролей вокруг стыка, история вместо перезаписи, основная БД — только PostgreSQL.

## Git — что важно помнить

Игнорируются: `.env`, `__pycache__`, `.venv`, `Работники/`, `tools/`, `.obsidian/`,
`desktop.ini`. Секреты и персональные данные в репозиторий не уходят.

Перед коммитом всегда проверять `git status` — в списке **не должно быть** `tools/`
и `Работники/`.

Типовой коммит выполняется в PowerShell из `D:\WeldPassport`:

```powershell
git status
git add -A
git commit -m "Краткое описание изменения"
```

## Ограничение среды Cowork

Агент работает с `D:\WeldPassport` напрямую: читать, создавать и редактировать файлы
может, а удалять и перемещать — нет (ограничение монтирования). Поэтому удаление,
перенос и git-коммиты готовятся командой PowerShell, которую запускает владелец.
Создание документов, правка кода и отчёты агент делает сам.

## Термины: база знаний и база данных — это разное

Названия похожи, но это два разных хранилища.

| | База знаний (Obsidian) | База данных (PostgreSQL) |
|---|---|---|
| Что хранит | Тексты, заметки, документы | Записи системы: стыки, сварщики, контроль, факт сварки |
| Что такое «сырьё» | Статьи, скриншоты, расшифровки, черновики | Исходные файлы для загрузки: выгрузки 1С по кадрам, табели, журналы сварки в Excel |
| Куда кладётся сырьё | `D:\Knowledge_Base\raw\` | На разбор — в `Работники/` или `tools/` (мимо git), затем импорт скриптами |
| Как обрабатывается | Превращается в wiki-страницы вручную/агентом | Импортируется в PostgreSQL скриптами `09_Разработка/scripts/` |
| Где находится | Вне репозитория (`D:\Knowledge_Base`) | PostgreSQL; файлы-источники в `Работники/`, `tools/` |

Коротко: «сырьё для базы знаний» — это заметки и статьи на разбор. Исходные данные
для приложения (выгрузки, табели) — это сырьё для **базы данных**, и идут они не в
`raw/`, а на импорт скриптами.

## Приём наработок (входящее)

Раскладывать новые материалы по папкам вручную не нужно — это делает агент.

1. Новый материал на разбор класть в `_Входящее/` (корень репо, игнорируется git —
   можно бросать черновики и выгрузки с ПДн) либо передать прямо в сообщении.
2. Одной фразой пояснить, что это и к чему относится.
3. Агент определяет место, оформляет файл в нужной папке, проставляет ссылки,
   обновляет навигацию и сообщает о дублях/конфликтах.
4. Перенос разобранного из `_Входящее/` и удаление — командой PowerShell от владельца.

Сырые материалы для базы знаний Obsidian — в `D:\Knowledge_Base\raw\`, не в `_Входящее/`.

## Типичный цикл работы

1. Сформулировать задачу.
2. Агент уточняет неоднозначное (по правилам `AGENTS.md` §6).
3. Агент читает `docs/` и `Context/`.
4. Выполнение: код и документы агент создаёт сам; удаление/перенос/коммит — командой
   владельцу.
5. Проверка результата.
